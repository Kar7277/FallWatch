## 今天踩过的坑 & 修复记录

### ❌ 错误 1：`ctx.cap` 是空指针，直接传给 `v4l2_init`

**问题：** 第一版代码 `capture_thread_ctx ctx = {0}` 后直接调 `v4l2_init(ctx.cap, ...)`，但 `ctx.cap` 是指针类型（`v4l2_capture_t *`），`{0}` 把它设成了 NULL。v4l2_init 内部第一行就写 `cap->device_path`，立即 segfault。

**根因：** 混淆了"指针"和"变量"——指针只是一张写着地址的纸条，`{0}` 把纸条清空成了 NULL，但没有任何真实的 `v4l2_capture_t` 对象存在。

**修复：** 栈上声明 `v4l2_capture_t cap = {0}`，然后 `ctx.cap = &cap`。`{0}` 提供干净初始状态，`v4l2_init` 负责填充所有字段。

**教训：** `v4l2_capture_t *cap` 只存地址（8 字节），真正的数据在别处。必须先有房子（变量），纸条（指针）才有意义。

### ❌ 错误 2：`ctx.rb` 忘了赋值

**问题：** 创建了 `rb = ring_buffer_init(8)`，但忘了 `ctx.rb = rb`。采集线程里 `ring_buffer_put(ctx->rb, frame)` 时 `ctx->rb` 是 NULL，空指针崩溃。

**修复：** 在 NULL 检查后、`v4l2_init` 前加 `ctx.rb = rb`。

**教训：** ctx 模式的每个指针字段都要显式赋值，`{0}` 不会替你填。

### ❌ 错误 3：v4l2_init 失败后调 v4l2_destroy 导致 double-free

**问题：** v4l2_init 在步骤 5/6 失败时已经调了 `free(cap->buffers)` + `close(cap->fd)`，但没重置 `buffers=NULL`、`buffer_count=0`、`fd=-1`。测试代码在 init 失败路径又调了 `v4l2_destroy`，导致 use-after-free 和 double-free。

**修复：** 两步——
1. v4l2_init 错误路径在清理后重置所有字段为安全值
2. test 代码在 init 失败后不再调 v4l2_destroy（init 自己清理干净了）

**教训：** 模块的 init 函数失败有两种清理策略：① init 自己全清，调用方只释放自己创建的资源；② init 留半成品，调用方调 destroy 清理。必须二选一并保持一致。本项目选了策略①。

### ❌ 错误 4：`v4l2_start` 返回值没检查

**问题：** `v4l2_start` 返回 `int`（0 成功，-1 失败），但第一版测试直接 `v4l2_start(ctx.cap);` 不检查，如果 STREAMON 失败，采集线程的 DQBUF 会永久阻塞。

**修复：** 加上 `if (v4l2_start(ctx.cap) != 0)` 错误处理，失败时调 `v4l2_destroy` + `ring_buffer_destroy` + return。

### ❌ 错误 5：消费者循环用 `sleep(1)` 而非连续轮询

**问题：** 第一版 `for (i=0; i<5; i++) { get; sleep(1); }`——每秒只取一次帧，中间 1 秒采集线程塞了 1~2 帧但没人取，缓冲区可能满导致旧帧被覆盖。

**修复：** 改成 `while(time < 10s) { get; if(f) { verify; free; } else { usleep(50000); } }`——连续轮询，没帧只等 50ms。

**教训：** 消费者应该用**连续轮询**而非**定时采样**，否则消费速度跟不上生产速度。

### ❌ 错误 6：判断逻辑写反 + 用了错误的变量

**问题：** `count > 0` 时报 Warning，`count == 0` 时报 SUCCESS 但打印 `ring_buffer_count(rb)`（此时恒为 0），显示 "SUCCESS: captured 0 frames!"——明明有帧却说 0。

**修复：** 用 `consumed`/`verified` 变量（实际计数）替代 `ring_buffer_count(rb)`（缓冲区残留数），并正确判断：`consumed == 0` → FAIL，`> 0` → PASS。

## 今天学到的新知识

### ⭐ 1. 指针 vs 变量的本质区别

这是今天最重要的认知升级。`v4l2_capture_t *cap` 和 `v4l2_capture_t cap` 完全不同：

| | 指针 `*cap` | 变量 `cap` |
|------|------|------|
| 本质 | 一张写地址的纸条（8 字节） | 真实的房子（~300 字节） |
| `{0}` 效果 | 纸条清空 = NULL | 房子清空 = 全零结构体 |
| 传参 | 传纸条（廉价） | 拷贝整栋房子（昂贵） |
| 能否直接给 init 用 | 不能——纸条指向 NULL | 能——init 往房子里填东西 |

### ⭐ 2. 防御性设计：错误路径字段重置

模块的 init 函数在失败时清理资源后，必须把字段重置为安全值（指针→NULL, fd→-1, count→0）。这确保了"无论什么状态调 destroy 都不会崩"——这是 C 语言手动资源管理的基本工程素养。

### ⭐ 3. 生产—消费轮询模式

消费者循环的正确写法是连续轮询（`while` + 短暂 `usleep`），而非定时采样（`for` + 长 `sleep`）。前者保证消费速度跟上生产速度，后者会导致积压和丢帧。

### ⭐ 4. drain 模式

生产者停止后，缓冲区可能还有残留数据。应该在 stop 后用 `while ((f = get()) != NULL)` 的 drain 循环全部取出，避免数据丢失，也验证"能否取到最后一帧"。

### ⭐ 5. 测试验证的判定逻辑

- `consumed == 0 && verified == 0` → FAIL（一帧都没采到，摄像头/驱动有问题）
- `consumed > 0 && consumed == verified` → PASS（全部帧数据完整）
- `consumed > 0 && consumed != verified` → FAIL（部分帧数据损坏）

不能只看 count 判断成败，要结合实际消费和验证结果。

### ⭐ 6. 集成测试的顺序

```
ring_buffer 自测 → v4l2_capture 自测 → capture_thread 集成测试
    (10/10)          (4/4 真机)          (228/228 真机)
```

自底向上、逐模块自测，集成时信心更足。如果集成测试失败，能迅速定位——因为底层模块已验证通过。

## 下午开发：motion_detect 模块设计与实现

### 1. motion_detect.h 设计（✅ 完成）

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 算法 | 三帧差法 | 二帧差法对快速运动敏感度不够（项目目标：检测老人摔倒） |
| Y 缓冲区数量 | 3 个（curr/prev1/prev2） | 无需存储差分结果，即算即弃 |
| 内存策略 | 堆上 malloc，结构体只存指针 | 结构体保持轻量，分辨率灵活可配 |
| 判定方式 | 两层过滤：单像素阈值 + 变化比例 | 先滤噪声像素，再滤小范围扰动 |
| 指针轮转 | 三指针循环右移 | 300KB 数据不动，只交换 3 个 8 字节指针 |

**`motion_detect_t` 结构体字段：**

| 字段 | 类型 | 用途 |
|------|------|------|
| `y_buffer_curr` | `unsigned char *` | 当前帧 Y 数据缓冲区 |
| `y_buffer_prev1` | `unsigned char *` | N-1 帧 Y 数据（三帧差法第一差） |
| `y_buffer_prev2` | `unsigned char *` | N-2 帧 Y 数据（三帧差法第二差） |
| `width` / `height` | `int` | 帧分辨率 |
| `threshold` | `int` | 单像素亮度差阈值（0~255） |
| `alarm_ratio` | `float` | 变化像素占比触发线（0~1） |
| `bg_frame_ready` | `int` | 背景帧就绪计数（<2 时不检测，只攒背景） |

**对外接口（3 个）：**

| 函数 | 返回 | 语义 |
|------|------|------|
| `motion_detect_init` | int | 0 成功 / -1 失败，含参数校验 + malloc 回滚 |
| `motion_detect_process` | int | 1=检测到运动 / 0=正常 / -1=错误 |
| `motion_detect_destroy` | void | 释放 3 个 Y 缓冲区，防 double-free |

`extract_y` 和帧差计算降为 `.c` 内 `static` 函数，外部不可见——单一职责原则。

### 2. motion_detect.c 实现（✅ 完成，4 轮审查）

**`extract_y` 内部函数：**
- 遍历 `total_pixels` 次，取 YUYV 偶数字节 `yuyv_data[i*2]`，写入 `dst[i]`
- YUYV 格式：每 4 字节 = Y₀, U, Y₁, V → Y 值隔字节分布

**`motion_detect_init` 防御性设计：**
- 参数校验：width/height ≤ 0、threshold 超范围、alarm_ratio 超范围均拒绝
- malloc 失败回滚：已分配的 buffer 逐个 free + 置 NULL
- 不 memset 清零——前两帧的 extract_y 会覆盖所有缓冲区，零值永不参与计算

**`motion_detect_process` 五步流程：**

```
1. 空指针 + data_size 校验
2. extract_y: frame->data → y_buffer_curr
3. if bg_frame_ready < 2: 指针轮转 + return 0（攒背景帧）
4. 三帧差循环:
     diff1 = |prev2[i] - prev1[i]|
     diff2 = |prev1[i] - curr[i]|
     if diff1>threshold && diff2>threshold: motion_pixels++
5. 指针轮转 → 判定: motion_pixels/total > alarm_ratio?
```

**指针轮转机制：** 三指针循环右移（tmp=prev2, prev2=prev1, prev1=curr, curr=tmp），每次调用 rotate 一次，数据零拷贝。

**`motion_detect_destroy`：** NULL 检查在最开头（防止空指针崩溃），逐个 free + 置 NULL。

### 3. 设计决策记录

#### 为什么选三帧差法而非二帧差法？

- 项目目标是检测老人摔倒，属于快速运动场景
- 二帧差法对快速移动物体敏感度不足（只比较相邻两帧）
- 三帧差法通过 |F₂-F₁| & |F₃-F₂| 取交集，既能捕获快速运动，又能抑制噪声
- 性能评估：每帧约 150 万次整数操作，Cortex-A7 @528MHz 绰绰有余

#### 为什么不缓存差分结果？

每个差分在相邻两次 process 调用中被重复计算一次（307,200 次减法）。评估：额外 CPU 开销约 0.3%，完全可忽略。缓存差分结果反而需要额外的 300KB 缓冲区 + 轮转管理逻辑，得不偿失。

#### 为什么用两层过滤（阈值 + 比例）？

单像素阈值过滤传感器噪声和光线抖动（微观层面），变化比例过滤小范围扰动如飞虫、窗帘（宏观层面）。两层各有分工，缺一不可。

### 5. 概念澄清

- **Edge 的含义：** 边缘计算指数据在产生地就地处理（摄像头→开发板→本地报警），不依赖云端，和用不用 AI 模型无关。帧差法跑在开发板上就是边缘计算。
- **帧差法 vs AI：** 帧差法是传统 CV，不算 AI。但架构设计上 `motion_detect_process` 是可替换模块——后续换成 TensorFlow Lite 模型推理，整体架构不动。面试时可以据此展示工程扩展性的考量。
- **Y 数据的本质：** Y 分量就是最普通的 `unsigned char` 数组，不需要任何库来定义。YUYV 里偶数字节就是 Y 值，隔字节提取。
- **process 为何无循环：** `process` 只做单帧判断，循环由更上层的**检测线程**负责——和 `ring_buffer_put`/`ring_buffer_get` 一样的模块化设计模式。
