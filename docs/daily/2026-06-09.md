摄像头支持 MJPG + YUYV 双格式。运动检测每秒 30 次需要 Y 分量，YUYV 下零开销；MJPEG 每帧都要解码 → CPU 扛不住。所以保持 YUYV，抓拍时才调 libjpeg-turbo 压缩。

## 今天踩过的坑 & 修复记录

### ❌ 错误 1：test 文件中 `mqtt_publisher_t *mq` 未初始化指针

- **问题**：声明为指针但未分配内存，传随机地址给 init → 段错误
- **根因**：跟 motion_detect 测试写 `motion_detect_t md` 栈分配的习惯不一致
- **修复**：改为 `mqtt_publisher_t mq;` 栈分配，传 `&mq`
- **教训**：指针和变量——之前踩过的坑又犯了

### ❌ 错误 2：`!mqtt_init(...)` 逻辑反转

- **问题**：init 成功返回 0，`!0` = true → 进入成功分支。语义晦涩
- **修复**：改为 `mqtt_init(...) == 0`
- **教训**：显式比较 `== 0` 比 `!` 取反可读性好

### ❌ 错误 3：mqtt_init 中 connect 失败后未 destroy 客户端

- **问题**：MQTTClient_create 成功后 create 返回已分配句柄，connect 失败时直接 return -1 导致句柄泄露
- **根因**：没有遵循"逐级回滚"原则（参照 sms_alert_init 每个错误路径都清理已创建资源）
- **修复**：connect 失败后调 `MQTTClient_destroy(&mq->client)` 再 return -1
- **教训**：多步初始化必须逐级回滚，参照 v4l2_init 的 rollback semantics

### ❌ 错误 4：paho 用系统 gcc 编译 → 链接失败

- **问题**：`.o` 是 x86-64 格式，交叉链接器报 "file format not recognized"
- **修复**：用 `arm-buildroot-linux-gnueabihf-gcc` 重编所有 paho 源文件
- **教训**：交叉编译环境要时刻注意用的是哪个 gcc

### ❌ 错误 5：bash `\` 换行在终端粘贴时失败

- **问题**：多行命令用 `\` 续行，粘贴后 `\` 后面空格导致参数被当成独立命令
- **修复**：写成一整行或确保 `\` 后紧跟换行符
- **教训**：给用户的命令优先给一整行

### ❌ 错误 6：`sizeof(指针)` 又犯了

- **问题**：`mq = malloc(sizeof(mq))` 中 mq 是指针，sizeof = 8 而非结构体大小
- **修复**：mqtt_init 是调用者提供内存模式（参照 gpio_alert_init），不需要内部 malloc
- **教训**：sizeof(指针) 永远=4/8，跟 sms_alert 第 4 个坑一模一样

### ❌ 错误 7：strcpy 当 strncpy 用了三个参数

- **问题**：`strcpy(mq->broker_uri, broker_uri, sizeof(mq->broker_uri))` 编译失败
- **根因**：strcpy 只有 2 个参数，strncpy 才有 3 个
- **修复**：改为 strncpy
- **教训**：C 标准库函数签名要记牢

### ❌ 错误 8：publish 括号位置放错

- **问题**：`MQTTClient_publish(..., (void*)json_msg), 1, 0, NULL` 右括号在 json_msg 后就把后面参数排除在外了
- **修复**：右括号移到最后一个参数 NULL 之后
- **教训**：函数调用的括号范围要仔细检查

### ❌ 错误 9：`sizepf` 写错了

- **问题**：`snprintf(cmdbuf, sizepf(cmdbuf), ...)` 编译失败
- **根因**：应该是 `sizeof`，不是 `sizepf`
- **修复**：改为 `sizeof(cmdbuf)`
- **教训**：打错字导致编译不过，IDE 静态检查可以提前发现

### ❌ 错误 10：`NUll` 大小写

- **问题**：`if(sms->fd != NUll)` 编译失败
- **根因**：C 区分大小写，NULL 必须全大写，`NUll` 是未定义标识符
- **教训**：NULL 全部大写

### ❌ 错误 11：free 后访问 sms->fd（use-after-free）

- **问题**：`free(sms); sms->fd = -1;` 释放内存后又写入
- **根因**：free 后指针悬空，访问悬空指针是未定义行为
- **修复**：删除 free 后的 `sms->fd = -1`，fd 重置已在 close 之后做了
- **教训**：free 永远是最后一步，之后不能再碰指针的任何成员

### ❌ 错误 12：destroy 里 NULL 检查顺序反了

- **问题**：先访问 `sms->fd` 再判 `sms != NULL`，如果 sms 是 NULL 直接崩
- **修复**：NULL 检查放在最前面：`if(sms != NULL && sms->fd != -1)`
- **教训**：指针判空永远是最外层 guard，先判空、后访问

### ⭐ 新认知：`free()` 不会把指针变成 NULL

- `free(sms)` 之后 sms 的值不变，变成"悬空指针"
- `if(sms != NULL)` 拦不住悬空指针，只能拦真正的 NULL
- 双重 destroy 安全需要调用方配合：`destroy(sms); sms = NULL; destroy(sms);`
- 这是 C 值传递的固有限制，改不了；glibc 的 tcache 能检测 double free 并 abort

## 今天学到的新知识

### ⭐ 1. MQTT 发布/订阅模型

- 三个角色：Publisher → Broker ← Subscriber
- 与 HTTP 请求/响应的本质区别：push vs pull
- Topic 分层命名（`/`），通配符 `+`（单层）`#`（多层）
- IoT 事实标准：AWS IoT、阿里云 IoT 全部基于 MQTT

### ⭐ 2. QoS 三级

- QoS 0：最多一次（可能丢，适合传感器周期性上报）
- QoS 1：至少一次（可能重复，告警选这个）
- QoS 2：恰好一次（开销大，适合计费系统）

### ⭐ 3. paho-mqtt C 库 5 个核心 API

- `MQTTClient_create` → `MQTTClient_connect` → `MQTTClient_publish` → `MQTTClient_disconnect` → `MQTTClient_destroy`
- `MQTTClient` = `void*` 句柄类型
- `MQTTClient_connectOptions_initializer` 宏初始化连接选项
- `MQTTCLIENT_PERSISTENCE_NONE` 不持久化（嵌入式内存紧张）

### ⭐ 4. Mosquitto broker

- 轻量 MQTT broker，一条命令启动：`mosquitto -d`
- `mosquitto_sub -t "topic" -v` 命令行订阅
- `mosquitto_pub -t "topic" -m "msg"` 命令行发布
- 默认端口 1883

### ⭐ 5. cmake 在 Windows 上交叉编译的坑

- Windows cmake 默认 VS generator，忽略 `-DCMAKE_C_COMPILER`
- 需要 `-G "MinGW Makefiles"` + toolchain file
- 最简单方案：在 Linux VM 上直接用交叉 gcc 手动编译

### ⭐ 6. 静态链接 vs getaddrinfo

- `-static` 完全静态链接时 glibc 的 NSS（域名解析）需运行时加载共享库
- 嵌入式 Buildroot 环境用动态链接更简单
