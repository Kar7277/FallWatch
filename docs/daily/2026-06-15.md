### 架构变更：单循环 → 双循环

```
while(g_running){                        // 外层：运动检测
    frame = ring_buffer_get(rb)
    YUYV→RGB + shm 写帧
    if(motion_detect_process(...)){
        SMS + MQTT 各发一次（上升沿）
        frame_free(frame)
        while(g_running){                // 内层：报警（不检测运动）
            frame = ring_buffer_get(rb)
            YUYV→RGB + shm 写帧
            LED 亮（状态变化才写 sysfs）
            warning=1
            JPEG 存图
            if(cancel 来了){
                LED 灭, warning=0
                frame_free(frame)
                break                  // 退出内层
            }
            frame_free(frame)
        }
        continue                        // 回到外层
    }
    frame_free(frame)
}
```

## 晚上：Cancel Alarm 按钮无效 + 帧率根因分析 + NEON 开工

### Cancel Alarm 按钮按下完全无反应

**排查过程**：

1. 追踪完整数据流 QT `onCancelAlarm()` → `cancelAlarm()` → 写 `cancel=1` 到 `/edgewatcher_events` shm → C 端 `shm_output_read_cancel()` 读 → break 内层。源码逻辑无问题。

2. 加 `qDebug()` 验证槽函数调用 → **终端无输出**，确认 `onCancelAlarm()` 根本没被调用。

3. 对比参考项目 `iot_sensor_test`（同板子、同 Buildroot Qt 5.12，按钮正常）发现：参考项目启动方式是：
   ```bash
   export QT_QPA_GENERIC_PLUGINS=tslib:/dev/input/event1
   export QT_QPA_PLATFORM=linuxfb:fb=/dev/fb0
   export QT_QPA_FONTDIR=/usr/lib/fonts/
   ```

4. **根因**：linuxfb 只负责帧缓冲渲染，不处理触摸输入。需要 **tslib 通用插件**（`libqtslibplugin.so`）而非 evdevtouch。之前试的 `-plugin evdevtouch` 是错的，Buildroot 的 Qt 5.12 通过 tslib 适配 i.MX 6ULL goodix-ts 触摸屏。

5. **修复**：按要求的环境变量启动后，Cancel Alarm 按钮正常响应。

### LED 初始亮

- **根因**：`gpio_alert_init()` 导出 GPIO 并设 `direction=out` 后未显式写初始值。内核默认 GPIO value=0，而 LED 是 active-low（0=ON），导致程序启动时 LED 立即亮。
- **修复**：在 `gpio_alert_init()` 末尾加 `gpio_led_off(gpio)`。

### 帧率 ~3fps 根因分析

1. 验证摄像头能力：`v4l2-ctl -d /dev/video1 --list-formats-ext` 确认 YUYV 640×480 支持 **30fps**，瓶颈在软件不在硬件。

2. 确认 `/dev/video0` 是 i.MX 6ULL 内部 PxP 处理器，真正 USB 摄像头在 `/dev/video1`。

3. 每帧耗时拆解：

| 步骤 | 运算量 | 估算耗时 |
|------|--------|----------|
| YUYV→RGB 逐像素转换 | 480×320 次循环 × 36 整数运算 ≈ 550万次 | **~14ms（大头）** |
| shm memcpy 900KB | CPU memcpy，DDR3 带宽 ~100MB/s | ~9ms |
| 运动检测 | 307K 字节 Y 分量三帧差法 | ~5ms |
| **合计** | | **~28ms → ~3.5fps** |

4. 解释为什么之前降采样到 320×240 帧率没变：只减了 shm memcpy（9→2ms），大头 YUYV→RGB 转换 14ms 纹丝未动。

5. **DDR3 带宽三重争抢**：
   - LCD 控制器：1024×600×32bpp × 60Hz ≈ 144MB/s 持续读
   - 摄像头 DMA：640×480×2 × 30fps ≈ 18MB/s 写
   - CPU：读帧 + 写 shm + QT 读 shm

6. **优化方案——NEON SIMD 重写 YUYV→RGB**：
   - 原理：用 ARM NEON 128 位 SIMD 指令，一条指令同时算 8 个像素
   - `vld4_u8` 一次加载自动解交织 Y/U/Y/V，`vmulq_n_u16` 并行乘法，`vqrshrn` 移位+窄化（自带 CLAMP）
   - 预计转换 14ms → 3ms，帧率翻倍到 ~6fps+
