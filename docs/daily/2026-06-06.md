## 今天踩过的坑 & 修复记录

### ❌ 错误 1：`vbufs[buf.index].start` 不存在

- **问题**：从 `test_uvc_lcd.c` 复制代码，用了不存在的局部变量 `vbufs`
- **根因**：复制代码没适配新上下文，忘了帧数据从 `frame->data` 来
- **修复**：改为 `frame->data`
- **教训**：跨文件搬代码时，先确认所有外部变量在新上下文里存在

### ❌ 错误 2：mmap 失败判断写 `== NULL`

- **问题**：`if(lcd->fb == NULL)` 判断 mmap 失败
- **根因**：和 malloc 的习惯搞混了
- **修复**：改为 `if(fb == MAP_FAILED)`
- **教训**：mmap 返回 `(void *)-1`，不是 NULL。这个坑 6/5 日记记过，今天又犯了——自己写的日记要回头看

### ❌ 错误 3：`=` 和 `==` 写错（致命级）

- **问题**：`if(rgb = NULL)` 把赋值当判断
- **根因**：手误，和 `fpirntf` 拼写错误同类型
- **修复**：改为 `if(rgb == NULL)`
- **教训**：手误无法避免，但这个是致命 bug——malloc 的指针被覆写成 NULL，内存泄漏 + 后续段错误。写完判断语句盯一眼等号数量

### ❌ 错误 4：frame 空指针检查在 malloc 之后

- **问题**：malloc 分配成功后检查 frame 为 NULL，return -1 时没 free rgb → 内存泄漏
- **根因**：没想到检查顺序和资源分配的关系
- **修复**：frame 检查失败时加 `free(rgb)` 再 return
- **教训**：函数开头先做便宜的参数校验，再做昂贵的资源分配——反过来就要每条错误路径都补 free

### ❌ 错误 5：`#ifndef GPIO_ALERT.H`

- **问题**：预处理器宏名含点号 `.`，不合法
- **根因**：复制 `GPIO_ALERT.H` 文件名当宏名
- **修复**：改为 `GPIO_ALERT_H`（下划线）
- **教训**：C 预处理器标识符只允许字母数字下划线，文件名习惯（点号）不能照搬

### ❌ 错误 6：ioctl 没检查返回值

- **问题**：FBIOGET_VSCREENINFO / FSCREENINFO 调用后直接使用 vinfo/finfo
- **根因**：从参考代码搬过来的，原代码在 main() 里也没检查
- **修复**：每个 ioctl 失败时 perror + close(fd) + return -1
- **教训**：参考代码不一定对，作为独立模块出错必须回滚

### ❌ 错误 7：结构体字段漏填

- **问题**：lcd_init 没填 xres/yres/bpp/line_length/src_width/src_height 等字段
- **根因**：只关注了"需要算的"字段（fb/fb_size/roff），忘了"直接赋值就行的"字段
- **修复**：逐个补全 12 个字段的赋值
- **教训**：写 init 函数时打开头文件对照着填，一个字段都别漏

## 今天学到的新知识

### ⭐ 1. LCD framebuffer 清屏用 memset

`memset(lcd->fb, 0, lcd->fb_size)` 一行清整屏。LCD 分辨率大于摄像头时，显存未覆盖区域是随机垃圾数据，启动时清一次保证干净黑背景，退出时清一次恢复黑屏。

### ⭐ 2. Linux sysfs GPIO 控制

/sys/class/gpio/export → 导出引脚；direction 设 out；value 写 1/0 控制高低电平。用 stdio (fopen/fprintf/fclose) 操作伪文件即可，比 POSIX 文件 API 更简洁。

### ⭐ 3. 参数校验顺序：便宜先做

函数开头先做指针非空检查（O(1)，几乎零开销），再做 malloc 堆分配（几百微秒）。反过来会让每条错误路径都多写一个 free，增加遗漏风险。

### ⭐ 4. C 预处理器标识符不能含点号

`GPIO_ALERT.H` 含 `.` → 编译不过。头文件保护宏用 `_` 替代文件名中的 `.`：`GPIO_ALERT_H`。
