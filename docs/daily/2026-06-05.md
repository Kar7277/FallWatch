## 今天踩过的坑 & 修复记录

### ❌ 错误 1：pthread 函数全部漏了 `&` 符号（~28 处）

- **问题**：`pthread_mutex_lock(rb->mutex)` — 函数要 `pthread_mutex_t *`，传了结构体值
- **根因**：不熟悉 pthread API，把 `rb->mutex` 当指针用了（它只是普通结构体字段）
- **修复**：全部改为 `pthread_mutex_lock(&rb->mutex)` 等
- **教训**：C 库函数取结构体字段地址是基本操作，看到编译报 `incompatible type` 优先检查是不是缺了 `&`

### ❌ 错误 2：init 中 return EXIT_FAILURE

- **问题**：`ring_buffer_init` 返回 `ring_buffer_t *`（指针），但错误路径写了 `return EXIT_FAILURE`（int 宏，值为 1）
- **根因**：从别处复制了错误处理代码，没改返回类型
- **修复**：全部改为 `return NULL`
- **教训**：复制错误处理代码时，**返回类型必须和目标函数一致**

### ❌ 错误 3：init 资源回滚不完整

- **问题**：cond_init 失败时只 `free(rb)`，之前初始化的 mutex 没销毁
- **根因**：错误路径只想着"退出"，没想着"清理已分配资源"
- **修复**：每步失败逐级回滚——not_full 失败 → destroy not_empty + mutex；not_empty 失败 → destroy mutex
- **教训**：多层初始化的回滚和 malloc 错误回滚是同一个模式——逐级销毁

### ❌ 错误 4：get_nonblock 里 tail 递增了两次

- **问题**：`rb->tail = (rb->tail+1) % rb->capacity;` 之后又 `rb->tail++;`
- **根因**：改了取模环回但忘删原来的 `tail++`
- **修复**：删掉多余的 `rb->tail++`
- **教训**：改完代码后逐行读一遍，确认没有残留的旧逻辑

### ❌ 错误 5：ring_buffer_count 有两个定义 + 死代码

- **问题**：旧版 count（无锁）未删除 → 链接重复定义；新版 count 里 `return` 在 `unlock` 前面 → 死代码
- **修复**：删旧版；新版用局部变量暂存 → unlock → return
- **教训**：改函数时要检查是否和原有定义冲突；锁内 return 会跳过 unlock

### ❌ 错误 6：fpirntf 拼写错误

- **问题**：`fpirntf`（字母顺序错），应为 `fprintf`
- **根因**：手误
- **教训**：编译报 `implicit declaration` 时先查拼写

## 今天学到的新知识

### ⭐ 1. pthread 条件变量的完整模式

经典阻塞队列的四要素：
1. `while (条件不满足 && active)` 等待（不是 if）
2. `cond_wait` 原子释放 mutex 并休眠，被唤醒后重新获取 mutex
3. 写操作后 `cond_signal` 通知等待者
4. `active=0` + `cond_broadcast` 实现优雅停止

### ⭐ 2. 优雅停止多线程的模式

停止顺序：**先停生产者 → drain 缓冲区 → 再停消费者**。用 `active` 标志 + broadcast 让所有阻塞在 cond_wait 上的线程醒来，检查 `!active` 退出。关键：`put` 在 `!active` 时丢弃数据返回（防御性），`get` 在 `!active && count==0` 时返回 NULL（正常 drain 终点）。

### ⭐ 3. LCD framebuffer 的 RGB 自动适配

不同 LCD 芯片的 RGB 字节顺序不同（XRGB vs XBGR），不能写死。运行时通过 `fb_var_screeninfo.red/green/blue.offset` 计算字节偏移，YUYV 转换时按偏移填入对应位置。`test_uvc_lcd.c` 已经是这样做的，提取模块时保留这个机制。

### ⭐ 4. mmap 失败返回 MAP_FAILED（不是 NULL）

`mmap()` 失败返回 `MAP_FAILED`（即 `(void *)-1`），判断时不能写 `if (!ptr)` 而要写 `if (ptr == MAP_FAILED)`。这是和 `malloc` 容易混淆的点。

### ⭐ 5. YUYV 色度上采样的含义

YUYV 4:2:2 中 U 和 V 是每 2 个水平像素共享的。RGB 转换时，两个 Y 各自算自己的 R/G/B，但共用同一组 U/V。这个逻辑在 BT.601 公式里隐含了——Y0 和 Y1 用同样的 `D=U-128, E=V-128`。
