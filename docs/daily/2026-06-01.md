## 今天踩过的坑 & 修复记录

### ❌ 错误 1：`munmap` 大小用像素数据算

**问题：** `destroy` 里 `munmap(cap->buffers[i], width * height * 2)`。驱动实际分配的 buffer 大小（含对齐）存在 `buf.length` 里，可能比像素数据大。

**修复：** 在头文件加 `size_t buffer_size` 字段，mmap 时把 `buf.length` 存进去，destroy 用它来 munmap。

**教训：** 内核分配的大小 ≠ 你算的大小。对齐、padding 都由驱动决定，必须用 `QUERYBUF` 返回的真实值。

---

### ❌ 错误 2：`cap->buffer_size = buf.length` 写在循环外

**问题：** `buf` 声明在 for 循环体内，出了循环作用域就没了。把 `cap->buffer_size = buf.length` 放循环外导致编译错误 `buf undeclared`。

**修复：** 放回循环内，mmap 成功后赋值。

**教训：** C99 的 for 循环内声明变量，作用域仅限于循环体。犯这种错说明对作用域还不够敏感。

---

### ❌ 错误 3：mmap 部分失败时只 `free` 不 `munmap`

**问题：** 循环里 mmap 第 i 个 buffer 失败时，索引 0~i-1 已成功映射但没被释放，直接 free + close 造成资源泄漏。QUERYBUF 失败和 QBUF 失败的同理，共 3 处。

**修复：** 抽取 `cleanup_mmap(cap, i)` 辅助函数，循环 munmap 已映射的 buffer，三处失败分支统一调用。

**教训：** 多步初始化中任何一步失败，必须回滚前面已完成的操作。称之为"回滚语义"（rollback semantics），是 C 语言资源管理的基本功。

---

### ❌ 错误 4：`mmap` 失败后对 `MAP_FAILED` 做 `munmap`

**问题：** 第一版修复中写了 `munmap(cap->buffers[i], ...)`，但此时 `cap->buffers[i] == MAP_FAILED`（即 `(void*)-1`），munmap 一个非法地址没有意义。

**修复：** 改为循环 munmap 索引 0 到 i-1（跳过当前失败的那个）。

**教训：** 失败的那个还没映射成功，清理的是前面"已经成功"的那些。

---

### ❌ 错误 5：`v4l2_destroy` 没先停止流

**问题：** 如果采集还在跑就直接 munmap，硬件可能在 DMA 写入已释放的内存，板子会死机或产生难以调试的内存损坏。

**修复：** destroy 开头先调 `v4l2_stop(cap)`，确保 `VIDIOC_STREAMOFF` 之后再释放。

**面试说法：** "资源释放必须遵循'先停后释放'的顺序——先停止数据流，保证 DMA 不再写入，再释放内存映射和文件描述符。"

---

### ❌ 错误 6：能力检查没区分两个 flag

**误解：** 以为 `V4L2_CAP_VIDEO_CAPTURE` 和 `V4L2_CAP_STREAMING` 是同一件事，判断了一个就不用判断第二个。

**澄清：** 它们是 `capabilities` 字段的**两个不同 bit 位**：
- `V4L2_CAP_VIDEO_CAPTURE`：这是个视频采集设备（不是广播、音频设备）
- `V4L2_CAP_STREAMING`：支持 mmap 流式传输（不是只能慢速 `read()`）

一个设备可能支持采集但不支持 streaming，两者必须分别检查。

## 今天学到的新知识

### ⭐ 1. V4L2 buffer 队列机制（核心概念）

这是今天最重要的收获。V4L2 用一组内核缓冲区实现摄像头和用户程序之间的**零拷贝数据传输**，通过两个队列管理：

```
           QBUF                         DQBUF
  用户 ──────────→ [待填充队列] ──→ [已完成队列] ←────────── 用户
                   (驱动往里写)       (等着你来取)
```

**队列中的 buffer 只有三种状态：**

```
  [空闲] ──QBUF──→ [等待填充] ──填满──→ [已完成] ──DQBUF──→ [用户持有]
    ↑                                                           │
    └──────────────────── QBUF ←────────────────────────────────┘
```

### ⭐ 2. QBUF（Queue Buffer）= 归还空柜子

**时机：** init 完成后（全部入队）、每帧处理完后（归还一个）

**含义：** 告诉驱动"这个 buffer 现在是空的，你可以往里写下一帧数据了"。buffer 从"用户持有"状态回到驱动的"待填充队列"。

**类比：** 你把空快递柜的钥匙交给快递员，告诉他"这个柜子归你用，有快递就往里放"。

### ⭐ 3. DQBUF（Dequeue Buffer）= 取出装满的柜子

**时机：** 采集循环中，需要帧数据时

**含义：** 从驱动的"已完成队列"中取出一个填满数据的 buffer。如果队列为空（没有准备好的帧），**阻塞等待**直到摄像头产生新帧。

**关键字段：**
- `buf.index`：第几个 buffer（0~3），对应 `cap->buffers[buf.index]` 的 mmap 地址
- `buf.bytesused`：驱动实际写入的字节数（640×480×2 = 614400 字节），验证帧有效性的关键指标
- `buf.timestamp`：帧的时间戳（`struct timeval`）

**类比：** 你去快递柜取件，柜子里有你买的 300KB 的"视频帧"包裹，取走后柜子又变空了。

### ⭐ 4. 为什么 DQBUF 和 QBUF 必须成对出现

只有 4 个 buffer。如果只 DQBUF 不 QBUF：

```
第1帧 DQBUF → 持有 buffer 0（不还）
第2帧 DQBUF → 持有 buffer 1（不还）
第3帧 DQBUF → 持有 buffer 2（不还）
第4帧 DQBUF → 持有 buffer 3（不还）
第5帧 DQBUF → 【永久阻塞！驱动没有可用的空 buffer 了】
```

所以循环必须是 `DQBUF → 处理 → QBUF` 这三个步骤的严格重复。

### 5. `mmap` 的本质（重温）

不是直接暴露物理地址，而是把内核空间的内存页映射到用户进程的虚拟地址空间，中间隔着 MMU 页表。V4L2 场景下省掉了 `read()` 时内核到用户的 614KB 拷贝（每帧）。

### 6. 回滚语义（rollback semantics）

多步初始化中任何一步失败，必须**逆序撤销**前面已完成的操作。这是 C 语言手动资源管理的基本要求，也是面试中体现工程素养的点。

### 7. 单元测试的隔离原则

测试 `v4l2_capture` 时不通过 `ring_buffer` 存取帧，因为：
- ring_buffer 已自测通过，是可信模块
- 如果测试失败，无法区分是 V4L2 问题还是 ring_buffer 问题
- 每个模块独立验证 → 集成时信心更足

### 3. 三个小问题修复

#### ① `cleanup_mmap` + `static` 的矛盾

**发现：** 头文件声明了 `void cleanup_mmap(...)`（公开），但 .c 里写了 `static void cleanup_mmap(...)`（内部）。两个声明互相矛盾。`static` 定义会覆盖，但如果有其他文件 include 头文件后调用该函数，链接时炸 `undefined reference`。

**决策：** `cleanup_mmap` 的 `int i` 参数语义是"回滚前 i 个已映射 buffer"，是 `v4l2_init` 的实现细节。保留头文件声明（公开 API），.c 去掉 `static`。

#### ② DQBUF 超时保护

**问题：** 摄像头是随时可能出问题的硬件（USB 被拔、死机、带宽被抢），DQBUF 在无数据时永久阻塞，线程会静默卡死——监控系统最糟糕的故障模式。

**解决：** 用 `poll()` 在 DQBUF 前检查文件描述符是否可读，设 2000ms 超时。

#### ③ `-Wall -Wextra` 编译标志

**`-Wall`：** 开启一大类常见警告（名字有误导性，不是"所有"警告）
**`-Wextra`：** 在 `-Wall` 基础上再开一批警告

**首次编译发现的 warning：** `ring_buffer_init` 调用签名不匹配——返回 `ring_buffer_t *` 但没接住返回值，且到处传 `&rb`（二级指针）给期望一级指针的函数。修复后 0 warning。

### 5. capture_thread 模块设计与实现（✅ 完成）

**架构角色：** 把 V4L2 和 RingBuffer 这两个独立模块连接起来，运行在独立线程中。

```
V4L2驱动              采集线程                   环形缓冲区
┌──────┐  DQBUF取帧  ┌────────────┐  put       ┌──────────┐
│ mmap │ ──────────→ │ 构造frame_t │ ─────────→ │ 4槽位    │
│ buf  │ ←────────── │ memcpy拷贝  │            │ 等着消费  │
└──────┘    QBUF归还  │ 然后QBUF    │            └──────────┘
                      └────────────┘
```

**新增文件：**

| 文件 | 说明 |
|------|------|
| `capture_thread.h` | `struct capture_thread_ctx`（cap/rb/thread/running/frame_id）+ 2 接口 |
| `capture_thread.c` | `capture_loop` 线程主循环 + `start`/`stop` |

**`capture_loop` 主循环流程（5 步）：**

```
while (running) {
    1. poll(fd, 2000ms)           → 超时/失败/POLLERR → break
    2. DQBUF 取出帧                → 失败 → break
    3. 构造 frame_t:
       - malloc frame_t           → 失败 → break
       - malloc frame_t->data     → 失败 → free(frame) → break（防泄漏！）
       - memcpy(buffers[buf.index], data, bytesused)
       - 填 id / data_size / timestamp（gettimeofday，毫秒精度）
    4. ring_buffer_put(rb, frame) → 塞进环形缓冲区
    5. QBUF 归还 buffer           → 失败 → break
}
```

**每步都有错误处理**——任何一个环节失败都有明确的 stderr 输出，故障定位不会抓瞎。

### 6. 今天学到的新概念

#### ⭐ 1. `_t` 后缀 = type（类型）

C 语言约定：`_t` 结尾的是自定义类型名。POSIX 保留了所有 `_t` 结尾的名字，但社区广泛使用，只要不和系统类型（`pid_t`、`pthread_t`、`time_t`）撞名就行。

#### ⭐ 2. `ctx` = context（上下文）

把模块相关的所有状态打包进一个结构体，所有函数通过指针传它——C 语言里最常用的"面向对象替代方案"。优势：新增字段函数签名不用改。

#### ⭐ 3. pthread 多线程基础

| API | 作用 |
|-----|------|
| `pthread_create(&tid, NULL, func, arg)` | 创建线程，arg 传给 func |
| `pthread_join(tid, NULL)` | 等线程结束 |
| `func` 签名：`void *func(void *arg)` | 参数和返回值都是 `void*` |
| **编译必须加 `-lpthread`** | 容易忘！ |

#### ⭐ 4. `volatile` 的必要性

线程 A 循环读 `ctx->running`，主线程在 `stop` 里写它。编译器 `-O2` 优化可能把循环里的读优化成"只读一次寄存器然后死循环"。`volatile` 告诉编译器"每次从内存读"。

#### ⭐ 5. `gettimeofday()` 毫秒时间戳

比 `clock_gettime` 简单，比 `time(NULL)` 精度高。头文件 `<sys/time.h>`。

#### ⭐ 6. V4L2 buffer index ≠ 环形缓冲区 index

**这是今天最重要的认识修正：** V4L2 的 `buffers[0~3]` 和 ring_buffer 的 `buffer[0~capacity-1]` 是两套完全独立的系统。DQBUF 返回的 `buf.index` 告诉你这一帧是哪个 V4L2 buffer（用于取 mmap 地址），和 ring_buffer 的 tail 毫无关系。

#### ⭐ 7. 内存泄漏防护模式

```c
a = malloc(...);     if (!a) → break;
b = malloc(...);     if (!b) → free(a) → break;  // 回滚前面成功的分配！
```

#### ⭐ 8. 嵌入式 Linux 文件系统

`/root` 通常是 tmpfs（RAM 文件系统），断电/重启文件全丢。永久保存需推到 Flash 分区（如 `/boot`）。
