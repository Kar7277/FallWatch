## 今天踩过的坑 & 修复记录

### ❌ 错误 1：TX/RX 同名对接

- **问题**：Air780E TX→J7 pin9 TXD，Air780E RX→J7 pin11 RXD，模块无回复
- **根因**：UART 必须交叉接——TX 对 RX，RX 对 TX。同名对接是两张嘴对吼
- **修复**：对调接线。TX→RXD，RX→TXD
- **教训**：TX=Transmit（嘴），RX=Receive（耳朵），永远用你的嘴对对方的耳朵

### ❌ 错误 2：`memset(sms->fd, 0, sizeof(sms->fd))` — 段错误

- **问题**：memset 第一个参数是 `sms->fd`（int 值），当作指针传给 memset
- **根因**：把结构体成员 fd 当成了结构体指针 sms，混淆了 `&sms->fd` 和 `sms`
- **修复**：`memset(sms, 0, sizeof(*sms))` 整结构体清零
- **教训**：memset 第一个参数必须是指针（地址），传 int 值 = 把随机数当地址访问 = crash

### ❌ 错误 3：`cfsetispeed(&sms, B115200)` — 传错结构体

- **问题**：cfsetispeed 要 `struct termios *`，传了 `sms_alert_t *`
- **根因**：两个结构体类型完全不一样，但 `void*` 隐式转换不报错，运行时会把业务结构体按 termios 布局写坏
- **修复**：改为 `cfsetispeed(&opt, B115200)`
- **教训**：termios 函数只接受 `struct termios *`，不能把业务结构体传进去

### ❌ 错误 4：`sizeof(uart_dev)` / `sizeof(phone)` — 指针大小

- **问题**：`uart_dev` 和 `phone` 是函数参数（指针），sizeof 拿到 4 字节而非缓冲区大小
- **根因**：跟 ring_buffer 开发时的错误（变量名暗示什么都 sizeof 什么）类似，函数参数指针 sizeof 永远返回指针宽度
- **修复**：改为 `sizeof(sms->uart_dev)` / `sizeof(sms->phone)`
- **教训**：sizeof(函数参数指针) = 4 或 8，不是数组大小；用 `sizeof(结构体成员)` 才正确

### ❌ 错误 5：`ECHO | ICANON` 写在 `c_cflag` 里

- **问题**：`opt.c_cflag &= ~(ECHO | ICANON | ...)`，ECHO/ICANON 是 c_lflag 的标志位
- **根因**：termios 有两组独立标志位，c_cflag（硬件控制）和 c_lflag（本地模式），混写了
- **修复**：拆成两行——c_cflag 关 CRTSCTS/PARENB/CSTOPB+设 CS8，c_lflag 关 ECHO/ICANON/ISIG
- **教训**：c_cflag=硬件层，c_lflag=行规程层，不能混用

### ❌ 错误 6：`resp[n] = '\n'` — 换行符写成了空字符

- **问题**：`\n`（0x0A）不是 `\0`（0x00），strstr 需要的是字符串终止符
- **根因**：手误，两个转义字符长得太像
- **修复**：改为 `'\0'`
- **教训**：`\n` = newline，`\0` = null terminator，差之毫厘、strstr 行为完全不一样

### ❌ 错误 7：`snprintf(buf, 32, ...)` 写字符串字面量

- **问题**：buf 指向字符串字面量（.rodata 只读段），snprintf 尝试写入→段错误
- **根因**：`char *buf = "AT+CMGF=1\r\n"` 指针指向只读内存，然后又用 snprintf 写同一个地址
- **修复**：声明可写数组 `char cmdbuf[64]`
- **教训**：字符串字面量在只读段，不能当写缓冲区；需要可写空间用 `char arr[N]`

### ❌ 错误 8：`*expected = "OK"` — 类型不匹配

- **问题**：`*expected` 是 `char`（解引用），`"OK"` 是 `char *`，赋值类型错误
- **根因**：想重新指向另一个字符串，但多写了一个 `*`
- **修复**：改为 `expected = "OK"`（去掉 `*`）
- **教训**：`*p = ...` 是改内容（需要同类型），`p = ...` 是改指向

### ❌ 错误 9：`resp[n] = '\0'` 放在 strstr 之后 / 放在 n 检查之前

- **问题**：三种错误变体——放在 strstr 之后（白写）、放在 n 检查之前（n=-1 时数组越界）、根本没写（strstr 搜垃圾）
- **根因**：read→null-terminate→strstr 三步顺序在 3 个位置反复写错
- **修复**：统一为 `read → n<=0 检查 → resp[n]='\0' → strstr`
- **教训**：read 原始字节不自动加 `\0`，这是 UART 编程和文件读取的本质区别

## 今天学到的新知识

### ⭐ 1. SIM 卡 vs 4G 模组的关系

- SIM 卡通信协议是 ISO 7816（CLK/RST/I/O/VCC），不是 UART
- SIM 卡必须物理插在 4G 模组上，因为基带芯片才能跟 SIM 卡对话
- 开发板上的 nano SIM 卡槽如果没有板载 4G 模组，只是一个空座子
- Air780E 自带基带 + SIM 卡槽，通过 UART 接受 AT 指令控制

### ⭐ 2. Air780E AT 指令发短信流程

三步：`AT+CMGF=1`（文本模式）→ `AT+CMGS="号码"`（设收件人等 `>`）→ 发内容 + `\x1A`（Ctrl+Z 结束）→ 等 `+CMGS: xxx OK`

### ⭐ 3. UART 交叉接线（TX↔RX）

- TX = 嘴（发送），RX = 耳朵（接收）
- 两设备相连：TX↔RX、RX↔TX，必须交叉
- 同名对接 = 两张嘴对吼，谁也没在听
- CTS/RTS/RST 在 AT 指令场景不需要接（关硬件流控即可）

### ⭐ 4. termios 两组标志位的区别

- `c_cflag`：硬件控制（波特率、数据位 CS8、校验 PARENB、停止位 CSTOPB、流控 CRTSCTS）
- `c_lflag`：行规程/本地模式（回显 ECHO、规范模式 ICANON、信号 ISIG）
- 不能交叉混写，必须各写各的

### ⭐ 5. VTIME / VMIN 控制 read 超时

- `VTIME=10, VMIN=0`：read 最多等 1 秒（10×0.1s），没数据就返回 0
- 不设的话串口 read 可能永久阻塞
- 这是非规范模式下的核心配置

### ⭐ 6. i.MX 6ULL UART 设备节点命名

- `ttymxcN` = IMX UART 驱动注册的设备（不在内核里不会出现）
- `ttySN` = 8250/16550 兼容串口驱动预分配端口（多为空壳）
- UART6 对应 `ttymxc5`（从 0 编号）

### ⭐ 7. const 关键字 + strstr 函数

- `const char *p`：承诺不通过 p 修改内容，编译器帮你检查
- `strstr(haystack, needle)`：在大串里搜小串，找到返回指针，找不到返回 NULL
- read() 不自动加 `\0`，必须手动加，否则 strstr 扫描越界

### ⭐ 8. 模块测试分层

- 有数据流转的模块（ring_buffer、v4l2、motion_detect、sms_alert）→ 值得写独立 test
- 纯 IO 操作模块（lcd、gpio）→ 集成时直接验证即可
