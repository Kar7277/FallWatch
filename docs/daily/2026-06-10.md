### jpeg_save_frame 实现方案梳理

**七步流程：**

```
① 参数校验        js / yuyv_data / out_path / path_len
② 生成文件名       time() + localtime() + strftime → "snap_20260610_143021.jpg"
③ 打开文件         fopen(out_path, "wb")
④ 配置压缩参数     JCS_RGB, 3 components, jpeg_set_quality, jpeg_stdio_dest
⑤ 逐行写入         for each row: YUYV行 → RGB行 → jpeg_write_scanlines
⑥ 完成压缩         jpeg_finish_compress
⑦ 清理             jpeg_destroy_compress + fclose + free → return 0
```

**核心知识：**

1. **YUYV → RGB 转换**：每 4 字节 YUYV（Y₀,U,Y₁,V）→ 6 字节 RGB，ITU-R BT.601 定点整数公式。U/V 两个像素共享，这是 YUV 4:2:2 的特点。

2. **setjmp/longjmp 错误处理**：libjpeg 默认出错 `exit(1)`，用自定义 error_exit + longjmp 跳回 setjmp 点，统一清理资源后 return -1。和 paho MQTT 的错误回调机制思路类似，但用的是 C 标准库的 longjmp。

3. **YUYV 数据来源确认**：数据流为 `/dev/video1 → V4L2 DQBUF → capture_thread 构造 frame_t → ring_buffer_put → ring_buffer_get → frame->data（YUYV 裸数据）→ motion_detect_process → 检测到运动 → jpeg_save_frame(js, frame->data, ...)`。

4. **调用者提供内存模式**：`jpeg_save_init` 是 `int init(jpeg_save_t *js, ...)` 签名，调用者栈分配结构体，destroy 不应 free 结构体。和 `gpio_alert_init` 同模式，和 `sms_alert_init`（内部 malloc）不同。

## 今天踩过的坑 & 修复记录

### ❌ 错误 1：int 参数跟 NULL 比较

- **问题**：`w == NULL || h == NULL || quality == NULL`，w/h/quality 是 int
- **根因**：习惯了指针判空，对数值参数也写了 NULL
- **修复**：改为数值范围检查 `w <= 0 || h <= 0 || quality < 1 || quality > 100`
- **教训**：NULL 是 `(void*)0`，只能用于指针。对 int 用 NULL 虽然大多数平台编译通过，但语义错误且 `-Wall -Wextra` 会报警

### ❌ 错误 2：数组名被当成左值赋值

- **问题**：`js->output_dir = dir`，output_dir 是 `char[256]`
- **根因**：数组名是常量地址，不能被赋值
- **修复**：`strncpy(js->output_dir, dir, sizeof(js->output_dir) - 1)`
- **教训**：C 里数组不是 first-class 类型，赋值/传参都会退化成指针

### ❌ 错误 3：free() 嵌入式数组 → 段错误

- **问题**：`free(js->output_dir)`，output_dir 是结构体内的 `char[256]`
- **根因**：free 只能释放 malloc/calloc/realloc 返回的堆地址。嵌入式数组在栈上或结构体内部，地址不属于堆
- **修复**：删除 free
- **教训**：free 之前必须确认这个指针是不是 malloc 来的。嵌入式数组的地址和 malloc 地址在虚拟地址空间的不同区域

### ❌ 错误 4：destroy 错误地 free 了调用者的结构体

- **问题**：`free(js)` — init 是调用者提供内存模式
- **根因**：混淆了两种 init 模式：`int init(T *obj)` → 调用者分配，不 free；`T* init(...)` → 内部 malloc，destroy 时 free
- **修复**：删除 free(js)，改为释放内部 libjpeg 资源
- **教训**：init 签名决定 destroy 行为 — 返回指针的才 free，接收指针的不 free

## 今天学到的新知识

### ⭐ 1. YUYV 内存布局与色彩下采样

- YUYV = YUV 4:2:2，每 2 个像素共享 1 对 UV
- 4:2:2 含义：每 4 个 Y → 2 个 U + 2 个 V（水平方向色度分辨率减半）
- 对比 4:2:0（JPEG 用）：水平+垂直都减半
- YUYV → JPEG 需要色度重采样，libjpeg 内部自动处理

### ⭐ 2. YCbCr → RGB 色彩空间转换

- BT.601 标准定义了标清数字视频的色彩矩阵
- 浮点公式 → 定点整数：系数 × 256 → `>> 8` 代替 `/256`
- `+128` 在移位前加上 = 四舍五入，减少截断误差
- 两个像素共用同一对 UV，简化了循环结构

### ⭐ 3. setjmp/longjmp 非局部跳转

- `setjmp` 保存当前 CPU 上下文（PC/SP/寄存器）到 `jmp_buf`
- `longjmp` 恢复上下文，"跳回"到 setjmp 位置，且 setjmp 返回非零值
- 类似 C++ 的异常机制，但无析构函数 → 必须手动清理资源
- libjpeg 用这个做错误恢复，因为 C 没有 try/catch

### ⭐ 4. libjpeg-turbo 的输入色彩空间

- `JCS_RGB` — 3 通道 R/G/B，最简单但多做一次色彩转换
- `JCS_YCbCr` — 需要分离的 Y/Cb/Cr 平面，效率高但复杂
- `JCS_GRAYSCALE` — 纯 Y 分量，体积最小但丢色彩
- 对嵌入式首版选 JCS_RGB：逻辑简单、好调试、编码开销在可接受范围

### ⭐ 5. C 数组与指针在结构体中的区别

| 声明 | 内存位置 | sizeof | 能否 free | 能否赋值 |
|------|---------|--------|-----------|---------|
| `char output_dir[256]` | 结构体内部 | 256 | ❌ | ❌ |
| `char *output_dir` | 指向外部 | 4/8 | ✅ | ✅ |

sms_alert 的 `uart_dev[32]`、gpio 的路径 `char[64]` 也都是嵌入式数组——整个项目都是这种风格。

### yuyv_to_rgb_row 实现要点

- **输入**：`yuyv_data`（每行 width×2 字节），**输出**：`rgb_row`（每行 width×3 字节）
- **循环**：`for(i = 0; i < width/2; i++)`，每次处理 4 字节 YUYV → 6 字节 RGB（2 个像素）
- **索引**：YUYV 读 `i*4+0~3`（Y₀,U,Y₁,V），RGB 写 `i*6+0~5`（R₀,G₀,B₀,R₁,G₁,B₁）
- **公式**：BT.601 定点整数，`298*C + 409*E + 128 >> 8` 模式，系数已含 ×256 缩放
- **CLAMP 宏**：`#define CLAMP(v) ((v) < 0 ? 0 : (v) > 255 ? 255 : (v))`
