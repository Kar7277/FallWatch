### 编译踩坑：libjpeg 版本不匹配

交叉编译了 libjpeg-turbo 3.1.4.1（产出 libjpeg.a v80），但工具链 sysroot 自带的 `jpeglib.h` 是 v62，头文件 struct 布局和库对不上 → 段错误。最终改用工具链自带 libjpeg（头+库版本一致），编译链接加 `-ljpeg` 即可，无需 `-L../lib`。

## 今天踩过的坑 & 修复记录

### ❌ 错误 1：jpeglib.h 报 `unknown type name 'FILE'`

- **问题**：`jpeglib.h` 使用了 `FILE*` 类型，但 `<stdio.h>` 未在其之前引入
- **根因**：`jpeg_save.h` 的 include 顺序——`<jpeglib.h>` 在 `<stdio.h>` 之前
- **修复**：在 `jpeg_save.h` 中 `#include <jpeglib.h>` 之前加 `#include <stdio.h>`
- **教训**：头文件 include 顺序很重要，依赖其他类型的头文件必须在被依赖者之后

### ❌ 错误 2：libjpeg 版本不匹配（library 62, caller 80）

- **问题**：运行时打印 "Wrong JPEG library version: library is 62, caller expects 80"，然后段错误
- **根因**：工具链 sysroot 的 `jpeglib.h` 是 v62，交叉编译的 `libjpeg.a` 是 v80，struct 布局不同
- **修复**：删除 inc/ 下手动复制的 jpeglib.h，编译时去掉 `-L../lib`，用工具链自带 libjpeg
- **教训**：交叉编译第三方库时，必须保证头文件和库文件版本一致。要么全部用自编译（头+库一起换），要么全部用工具链自带

### ❌ 错误 3：`js->initialized = 0` 写在 NULL 检查内部

- **问题**：`if(js==NULL) { js->initialized = 0; return -1; }` → NULL 指针解引用
- **根因**：把初始化代码放在了 NULL 保护的代码块里面
- **修复**：挪到 NULL 检查之后、参数检查之前
- **教训**：NULL 检查块内部不能解引用被检查的指针

## 今天学到的新知识

### ⭐ 1. CMake 交叉编译工具链文件（toolchain.cmake）

```cmake
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR arm)
set(CMAKE_C_COMPILER arm-buildroot-linux-gnueabihf-gcc)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
```

- `PROGRAM NEVER`：编译期工具（如代码生成器）用主机版本
- `LIBRARY/INCLUDE ONLY`：库和头文件只从交叉编译器 sysroot 找

### ⭐ 2. libjpeg 头文件/库版本必须一致

libjpeg 在运行时校验 `JPEG_LIB_VERSION`，头文件和库编译时的版本号不同会导致结构体布局错位 → 段错误。要么全部自编译，要么全部用工具链自带。

### ⭐ 3. 双 destroy 保护模式

给结构体加 `int initialized` 字段，init 成功置 1，destroy 检查 `==1` 才执行清理并重置为 0。这是 gpio_alert_init 模式之后又一个采用此模式的模块。
